# SmartChildcare Agent 3 分钟答辩稿

## 一句话版本

SmartChildcare Agent 把托育场景里最碎片、最难闭环的“教师记录、AI 会诊、园长决策、家长理解、家长执行反馈”做成了一个移动端优先的 Agent 工作流系统。

## 3 分钟版

### 0:00 - 0:20 开场

- “托育场景最难的不是有没有数据，而是老师没有时间整理、园长不知道先处理谁、家长不知道今晚具体做什么。”
- “我们做的不是一个托育后台，而是一个移动端优先的 SmartChildcare Agent。”

### 0:20 - 0:45 为什么是 Agent

- “因为托育工作不是一次性问答，而是一条连续工作流。”
- “它需要记录、判断、建议、执行、反馈、复查不断接力。”
- “所以我们用三端闭环来设计，而不是只做一个 AI 聊天框。”

### 0:45 - 1:05 Teacher 语音主线

- “老师先用语音入口把现场观察快速变成结构化草稿。”
- “这一步解决的是低成本记录，而不是让老师再多填一层表单。”

### 1:05 - 1:35 第一展示位：高风险会诊

- “如果系统识别到高风险，就进入多 Agent 会诊。”
- “会诊不是只出一段文字，而是直接给出教师动作、家长今夜任务和园长决策卡。”
- “这条链路是我们当前最强的 Agent workflow。”

### 1:35 - 1:55 第二展示位：Admin 决策区

- “园长端不再只是看报表，而是优先看今天最该处理的会诊和风险优先级。”
- “这让会诊结果真正进入机构级决策，而不是停留在教师页面。”

### 1:55 - 2:20 家长 wow factor：微绘本

- “家长首页不是冰冷的报表，而是把成长亮点、会诊和今晚任务压缩成 3 幕微绘本。”
- “它先解决的是愿不愿意看、看不看得懂。”

### 2:20 - 2:40 家长闭环：趋势线与反馈

- “看完故事后，家长还可以继续追问 7、14、30 天趋势，并在今晚执行后直接反馈。”
- “这样老师第二天不是重新开始，而是继续沿着上一轮上下文往下走。”

### 2:40 - 3:00 vivo 映射与保守边界

- “我们把 vivo 能力映射到真实产品入口，而不是只停留在概念上。”
- “例如 Teacher 语音入口对应 ASR，Multi-Agent 会诊对应 LLM，Parent 微绘本对应 storybook provider 方向。”
- “同时我们保留保守边界：当前只宣称代码层接入与 smoke/test/演示基础，不夸大成 fully live；staging 也仍处于最终 freeze 前收口阶段。”

## 为什么当前口径必须保守

- staging 还不能写成 `fully healthy / fully switched`
- `vivo_llm` / `vivo_asr` / `vivo_tts` / storybook media provider 不能写成 fully live
- Parent trend 必须依赖 FastAPI brain
- Admin 第二展示位不应扩写成 `T9D` / `T9C` 主战场已完成

## 评委可能会问的问题

### Q1：这是不是一个普通托育后台？

- 不是。核心不是报表，而是三端连续工作流。

### Q2：为什么要做 Multi-Agent？

- 因为高风险儿童处理不是单一问答，需要健康观察、饮食行为、家长沟通和园内动作协同输出。

### Q3：微绘本是不是只是包装？

- 不是。它承担的是“让家长愿意看、看得懂、接得住今夜任务”的入口，再由趋势线和反馈把行动闭环补全。

### Q4：现在是不是已经 fully live？

- 不是。我们刻意保守：当前可确认的是代码层接入、测试和演示链路已具备，staging 和真实上游仍在收口。

### Q5：为什么这套系统适合比赛？

- 因为它能在几分钟内讲清楚问题、路径、角色闭环和 AI 能力映射，同时保留工程可继续推进的基座。

## 收尾一句话

- “所以这套作品的核心价值不是多一个 AI 功能，而是把托育场景里最难闭环的一条链真正串起来，并且在 freeze 前把录屏、答辩、交接都收束到了同一套保守口径上。”
